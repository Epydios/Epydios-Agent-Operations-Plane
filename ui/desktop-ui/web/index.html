Instructions

Use Home as the living orientation surface. Keep workflow reminders, storage rules, and future doc links here so the rest of the app stays focused on active work.

Daily Operator Flow
The normal path should stay simple: do live work in Agent, move finished evidence review to History, and only use Incidents when a run needs durable downstream handoff.
  1. Start in Agent for the active thread, approvals, and thread state.
  2. Use History for past runs, evidence review, JSON drill-in, and artifact folder guidance.
  3. Use Incidents when you need a filing package, queue tracking, or handoff output.
  4. Use Settings for routing, storage, diagnostics, and runtime configuration changes.
Tab Roles
  • Home: living instructions, workflow reminders, and storage rules.
  • Agent: current chat, current approvals, and current thread state.
  • History: past runs, evidence, exported JSON, and folder organization.
  • Incidents: incident packages, queue state, audit-backed handoffs, and response exports.
  • Settings: provider routes, credentials, diagnostics, retention, and integration state.
Storage and Evidence Rules
Keep repo-safe provenance in Git-tracked paths. Keep screenshots, manual QA, launch snapshots, and bulky evidence in non-repo storage.
  • Repo folder stays clean for code, config, and lightweight provenance that should ship to GitHub.
  • Non-repo folder is the default home for screenshots, local runtime output, visual QA, and incident handoff artifacts.
  • Organize new evidence by date bucket first, then by run or package ID when the artifact is specific to one workflow.

Settings Workflow
Use this workspace to change configuration deliberately and verify the resulting control-plane state.
  1. Select the active agent profile, then open the relevant settings subpanel.
  2. Review configuration and diagnostics before applying a change.
  3. Use Open Audit Events after changes to confirm the recorded control-plane trail.

History Workflow

          
Use this tab for past activity, evidence drill-in, JSON review, and folder guidance. Current approvals stay in Agent.

          
    
            
  1. Apply scope, status, and time filters to narrow the run set.
    2. 
            
  2. Select a run row and review summary, evidence, and raw JSON in that order.
    3. 
            
  3. Use the artifact access section in run detail to copy repo and non-repo storage roots plus the current run date bucket.
    4. 
            
  4. Use Incidents for packaging and handoff after you finish the evidence review.
    5. 
          

        

        

          

            
Artifact Organization

            
Use History to keep run evidence predictable. The working convention is date bucket first, then the specific run or incident package under that day.

            
    
              
  • Put run-specific notes and screenshots under history/YYYYMMDD/run-id/.
    • 
              
  • Keep manual comparison captures in the readiness area when they are not specific to one runtime record.
    • 
              
  • Use the run detail Artifact Access section to copy the exact roots and suggested folders before leaving this tab.
    •

Incidents Workflow

          
Use this workspace when a run needs durable incident packaging, queue tracking, or audit-backed handoff output.

          
    
            
  1. Use queue filters or quick views to isolate drafted, filed, or closure-required packages.
    2. 
            
  2. Select packages, update filing status, and export incident JSON or copy the matching handoff summary.
    3.

    4. 
          

        

        

          

            
Queue Discipline

            
    
              
  • Drafted: package is prepared locally and still needs downstream handoff.
    • 
              
  • Filed: package has been handed off and is waiting on follow-through or closure.
    • 
              
  • Closed: queue tracking is complete unless new follow-up forces a reopen.
    • 
            

          

          

            
Audit Events Role

            
    
              
  • Use Audit Events when you need source-of-truth records, exportable audit evidence, or a fresh incident package.
    • 
              
  • Use the queue when a package already exists and you are tracking filing state, handoff, or closure.
    • 
              
  • Keep the handoff preview in view before copying or exporting anything downstream.
    •
